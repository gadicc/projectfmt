## Execution protocol

1. Run one plan at a time in numeric order unless the operator explicitly
   authorizes parallel execution after confirming that scopes do not overlap.
2. Before starting, require every listed dependency to be DONE, record the
   current HEAD, and change this plan's row to IN PROGRESS. A dependency's Go or
   Revise decision may require a new implementation plan before a downstream
   plan can start.
3. Treat each plan's excerpts and `5ff23b4` diff as the audit snapshot, not as a
   patch target. Re-read live files after all earlier work. Expected changes
   from completed plans are rebase inputs; stop only if they invalidate the
   finding, contract, or scoped approach.
4. Never accept a targeted test command that runs zero tests. For nested BDD
   cases in main_test.ts, run the complete file or one of its top-level suites
   (`resolveFormatter` or `formatSource`) and confirm the summary reports tests.
5. `plans/README.md` is an administrative exception to every plan's file scope:
   edit only that plan's status row and optional one-line blocked reason.
6. Finish by running the plan's gates, inspecting the complete diff, updating
   the status row, and committing only its stated scope plus the administrative
   index edit.

## Shared repository constraints

- Public behavior remains source-string, intended-file-path, and
  project-boundary first.
- Discovery and implementation resolution never cross `projectRoot`.
- Availability never changes a valid selection, and formatter failures never
  silently fall through.
- Prettier and Deno remain format-only. Biome mirrors configured `check --write`
  without unsafe fixes and retains `formatOnly`.
- Tests do not install packages or use the network at runtime.
- Generated `npm/` output is never hand-edited or committed.
- Use named ESM exports and keep Deno and generated Node behavior aligned.

## Findings considered and rejected

- Preserve workspace-manifest I/O failures during root inference: valid but
  lower-frequency than the selected boundary work; reconsider after plans 001
  and 011 because their shared path helpers may change the evidence.
- Return successful formatter stderr in `FormatSourceResult`: additive but no
  demonstrated user need; failed stderr is already preserved.
- Bound inferred-root cache growth: growth is real, but exploitability depends
  on an untrusted-path service use case not established by repository intent.
- Characterize in-flight root-cache invalidation: worthwhile regression
  coverage, but no current failure was reproduced.
- Add a generic upstream-compatibility matrix: its concrete cases are already
  assigned to plans 004–007; a separate umbrella plan would duplicate them.
- Build the npm artifact once per full gate: measurable duplication, but lower
  leverage than runtime per-call work and easy to revisit independently.
- Add persistent formatter/config caches: rejected because the README explicitly
  promises fresh evidence, configuration, and availability across calls.
- Build a whole-repository formatting CLI: rejected because the roadmap says it
  is not the primary goal; plan 016 stays diagnostic-only.
- Treat symlink traversal as a sandbox vulnerability: rejected because the trust
  model explicitly says `projectRoot` is not a sandbox. Plan 011 fixes the
  concrete false-unavailability bug only.
- Upgrade dependencies: no plan; the frozen audit found no high-severity
  advisory and the latest-version check found no outdated dependency.
